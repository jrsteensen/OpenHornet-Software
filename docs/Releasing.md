# Releasing OpenHornet Software

[TOC]

This page is for repository maintainers who publish OpenHornet Software release packages.

Releases are created from Git tags. Pushing a tag named `v*` starts the CI workflow, builds all firmware, builds local Doxygen HTML, creates a release zip, and publishes a GitHub Release.

---

## Release Tag Format

Use semantic version tags with a leading `v`:

```bash
v1.0.0
v1.4.2
```

Only tags that begin with `v` trigger the release package job.

---

## Create a Release

1. Make sure `main` contains the exact commit you want to release.
2. Make sure the normal CI and documentation checks are passing on `main`.
3. Update your local checkout:

```bash
git checkout main
git pull origin main
```

4. Create and push the release tag:

```bash
git tag v1.0.0
git push origin v1.0.0
```

Replace `v1.0.0` with the version you are releasing.

---

## What GitHub Actions Does

When the tag is pushed, `.github/workflows/ci.yaml` runs on that tag.

The workflow:

1. Checks out the repository with submodules.
2. Builds every sketch under `embedded/`.
3. Runs `make release` to collect flashable firmware files.
4. Uploads the `firmware` workflow artifact.
5. Builds Doxygen HTML from `docs/Doxyfile`.
6. Assembles `OpenHornet-Software-<tag>.zip`.
7. Publishes a GitHub Release and attaches the zip.

The release package contains:

- `firmware/` - `.hex`, `.eep`, and `.bin` files generated by CI.
- `embedded/` - embedded sketch source.
- `libraries/` - Arduino libraries and submodules.
- `include/` - build support and board Makefiles.
- `docs/` - generated Doxygen HTML, starting at `docs/index.html`.
- `README.md` - package-specific release README.
- `LICENSE.md` - project license.

---

## Release Notes

Release notes are generated by GitHub using the repository's release-note configuration in `.github/release.yml`.

GitHub compares the new release tag against the previous release and groups merged pull requests into categories such as firmware, documentation, bug fixes, and maintenance. Pull requests labeled `ignore-for-release` are excluded.

The release workflow also prepends a short note that tells users which zip to download and links to the public documentation site.

---

## Verify the Release

After the workflow succeeds:

1. Open the repository's **Releases** page.
2. Confirm the new release is published, not draft.
3. Confirm `OpenHornet-Software-<tag>.zip` is attached.
4. Download and extract the zip.
5. Confirm these paths exist:

```text
firmware/
embedded/
libraries/
include/
docs/index.html
README.md
LICENSE.md
```

6. Confirm `firmware/` contains `.hex`, `.eep`, and `.bin` files.
7. Open `docs/index.html` locally in a browser and confirm the Doxygen docs load.

---

## Retry a Failed Release

If the tag workflow fails before publishing a release, fix the problem, delete the local and remote tag, then recreate it on the corrected commit:

```bash
git tag -d v1.0.0
git push origin :refs/tags/v1.0.0
git checkout main
git pull origin main
git tag v1.0.0
git push origin v1.0.0
```

If a GitHub Release was already created, delete the failed release from the GitHub Releases page before reusing the tag.

Use the same version tag only when retrying the same intended release. For a new release after users may have downloaded the package, create a new version tag instead.
